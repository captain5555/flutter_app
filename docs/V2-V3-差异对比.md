# V2 vs V3 系统结构差异对比

## 一、目录结构对比

### V2 (简单架构)
```
nas-material-manager-v2/
├── backend/src/
│   ├── server.js              # 单文件入口+路由+上传逻辑
│   ├── db/
│   │   └── database.js        # 单文件SQLite回调API
│   └── routes/
│       ├── materials.js
│       ├── users.js
│       └── ai.js
└── frontend/public/
    └── index.html             # 127KB 单文件（HTML+CSS+JS）
```

### V3 (模块化架构)
```
nas-material-manager-v3/
├── backend/src/
│   ├── server.js              # 简洁入口
│   ├── config/

│   │   ├── database.js        # 数据库抽象工厂
│   │   └── storage.js         # 存储抽象工厂

│   │   ├── auth.js            # 认证路由
```

## 二、后端差异

### 2.1 数据库差异

| 项目 | V2 | V3 |
|------|-----|-----|
| **ID类型** | 字符串ID (`user1`, `admin`) | 整数ID (1, 2, 3, 4) |
| **用户表字段** | `id`, `name`, `password`, `role` | `id`, `username`, `password_hash`, `role` |
| **素材表字段** | `filename`, `oss_key` | `file_name`, `file_path`, `thumbnail_path` |
| **删除标记** | `deleted_at` (DATETIME, NULL=未删除) | `is_deleted` (BOOLEAN) + `deleted_at` |
| **额外表** | - | `folders`, `operation_logs` |
| **API风格** | Callback回调 | Promise/async-await |
| **密码存储** | 明文 | bcrypt哈希 |

### 2.2 API响应格式差异

**V2格式：**
```javascript
// 成功
{ id: 1, name: "user1", ... }
[ {id: 1, ...}, {id: 2, ...} ]

// 失败
{ error: "错误信息" }
```

**V3格式：**
```javascript
// 成功
{ success: true, data: { id: 1, ... } }
{ success: true, data: [ ... ] }

// 失败
{ success: false, error: "错误信息" }
```

### 2.3 认证差异

| 项目 | V2 | V3 |
|------|-----|-----|
| **认证方式** | URL参数 `?user_id=xxx` | JWT Bearer Token `Authorization: Bearer <token>` |
| **登录接口** | `POST /api/users/login` | `POST /api/auth/login-simple` |
| **Token内容** | 无 | `{ id: 1, username: "admin", role: "admin" }` |

### 2.4 路由对比

| 功能 | V2路由 | V3路由 |
|------|--------|--------|
| 登录 | `POST /api/users/login` | `POST /api/auth/login-simple` |
| 获取用户 | `GET /api/users` | `GET /api/users` |
| 创建用户 | `POST /api/users` | `POST /api/users` |
| 删除用户 | `DELETE /api/users/:id` | `DELETE /api/users/:id` |
| 获取素材 | `GET /api/materials/user/:userId/folder/:folderType` | `GET /api/materials/user/:userId/folder/:folderType` |
| 获取垃圾箱 | `GET /api/materials/user/:userId/trash?all=true` | `GET /api/materials/user/:userId/trash?all=true` |
| 上传素材 | `POST /api/materials/upload` | `POST /api/materials/upload` |
| 更新素材 | `PUT /api/materials/:id` | `PUT /api/materials/:id` |
| 批量删除 | `POST /api/materials/batch/trash` | `POST /api/materials/batch/trash` |
| 批量恢复 | `POST /api/materials/batch/restore` | `POST /api/materials/batch/restore` |
| 批量永久删除 | `DELETE /api/materials/batch` | `DELETE /api/materials/batch` |
| 批量复制 | `POST /api/materials/batch/copy` | `POST /api/materials/batch/copy` |
| 批量移动 | `POST /api/materials/batch/move` | `POST /api/materials/batch/move` (需 `targetFolder`) |
| 下载视频 | - | `GET /api/materials/:id/download` |
| AI设置 | `GET /api/ai/settings` | `GET /api/ai/settings` |
| 保存AI设置 | `PUT /api/ai/settings` | `PUT /api/ai/settings` |
| AI生成标题 | `POST /api/ai/generate-title` | `POST /api/ai/generate-title` |
| AI生成描述 | `POST /api/ai/generate-description` | `POST /api/ai/generate-description` |
| AI翻译 | `POST /api/ai/translate` | `POST /api/ai/translate` |

## 四、迁移策略

### 方案A：纯V3架构（推荐）
1. 保持V3后端架构不变
2. 创建完整的 `api-adapter.js` 转换层
3. 使用V2的完整前端代码，通过adapter连接V3后端
4. 逐步将V2前端重构为V3的模块化结构

### 方案B：混合架构
1. 后端同时支持V2和V3两种API格式
2. 前端使用V2的完整实现
3. 后续逐步迁移

### 方案C：完全重写前端
1. 保持V3后端
2. 按照V3的模块化架构重写所有前端功能
3. 工作量最大，但代码质量最高
